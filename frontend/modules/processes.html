Processes

Introduction to Processes and Process Attributes

What is a Process?

A process is simply an instance of one or more related tasks (threads) executing on your computer. It is not the same as a program or a command. A single command may actually start several processes simultaneously. Some processes operate independently, while others are related to each other. When one process fails, it may or may not affect other processes running on the system.

Process Types

A terminal window is a process that runs as needed, allowing users to execute programs and access resources interactively. Programs can also run in the background, detached from the shell.

Processes come in different types based on their tasks.

Process TypeDescriptionExamples
Interactive ProcessesNeed to be started by a user, either at a command line or through a graphical interface such as an icon or a menu selection.bash, firefox, top, Slack, Libreoffice
Batch ProcessesAutomatic processes which are scheduled from and then disconnected from the terminal. These tasks are queued and work on a FIFO (First-In, First-Out) basis.updatedb, ldconfig
DaemonsServer processes that run continuously. Many are launched during system startup and then wait for a user or system request indicating that their service is required.httpd, sshd, libvirtd, cupsd
ThreadsLightweight processes. These are tasks that run under the umbrella of a main process, sharing memory and other resources, but are scheduled and run by the system on an individual basis. An individual thread can end without terminating the whole process and a process can create new threads at any time. Many non-trivial programs are multi-threaded.dconf-service, gnome-terminal-server
Kernel ThreadsKernel tasks that users neither start nor terminate and have little control over. These may perform actions like moving a thread from one CPU to another, or making sure input/output operations to disk are completed.kthreadd, migration, ksoftirqd

Process Scheduling and States

A critical kernel function called the scheduler continuously shifts processes on and off the CPU. It allocates time based on relative priority, required processing time, and how much time has already been granted to each task.

When a process is in the running state, it is either actively executing instructions on a CPU or waiting for its allocated time slice to execute. All running processes exist in a run queue, with separate queues for each CPU or core in multi-processor systems. The term "running" can be somewhat misleading, as a process in this state may actually be temporarily swapped out, waiting for its turn to execute.

Process and Thread IDs

The operating system tracks multiple simultaneous processes by assigning each a unique process ID (PID). This identifier monitors process state, resource usage, and memory allocation.

PIDs are typically assigned sequentially as processes are created, with PID 1 reserved for the init process (system initialization).

ID TypeDescription
Process ID (PID)Unique Process ID number.
Parent Process ID (PPID)Process (Parent) that started this process. If the parent dies, the PPID will refer to an adoptive parent; on modern kernels, this is kthreadd which has PPID=2.
Thread ID (TID)Thread ID number. This is the same as the PID for single-threaded processes. For a multi-threaded process, each thread shares the same PID, but has a unique TID.

Terminating a Process

When an application malfunctions, you may need to terminate it.

To kill a process, use kill -SIGKILL <pid> or kill -9 <pid>.

You can only terminate your own processes unless you're root. Despite its name, kill can send various signals to processes, not just termination signals.

user@lfs101: ~
$ ps
    PID TTY          TIME CMD
  17599 pts/1    00:00:00 bash
  17775 pts/1    00:00:00 ps
  17785 pts/1    00:00:00 cat
$ kill -9 17785
[+1]  Killed          cat
$ ps
    PID TTY          TIME CMD
  17599 pts/1    00:00:00 bash
  17775 pts/1    00:00:00 ps

User and Group IDs

Multiple users can simultaneously access a system, each running various processes. The system tracks the process starter through their Real User ID (RUID).

Access rights are determined by the Effective UID (EUID), which may differ from the RUID.

Process Metrics and Process Control

Load Average

The load average measures system demand over time, counting processes that are:

  • Running on a CPU
  • Waiting in the run queue
  • Waiting for resources (typically I/O)

NOTE: Linux uniquely includes uninterruptible sleeping processes in this calculation.

View load average using w, top or uptime.

user@lfs101: ~
$ w
14:05:07 up  3:10,  1 user,  load average: 1.19, 1.67, 1.54
USER     TTY      FROM             LOGIN@   IDLE   JCPU   PCPU  WHAT
username  tty2     -                10:54   22:02   0.04s  0.04s /usr/libexec/gn
user@lfs101: ~
$ uptime
14:05:34 up  3:11,  1 user,  load average: 0.92, 1.57, 1.51

Interpreting Load Averages

Load average shows three numbers representing system utilization over different time periods. For a single-CPU system:

  • 0.45: 45% utilization during the last minute
  • 0.17: 17% utilization over the last 5 minutes
  • 0.12: 12% utilization over the last 15 minutes

Background and Foreground Processes

In Linux, a command or "job" can run in two ways:

  • Foreground: The default way. It takes over your terminal, so you must wait for it to finish before you can type another command.
  • Background: The command runs behind the scenes, letting you continue to use the terminal. This is useful for tasks that take a long time.

The basic controls:

  • command &: Starts a job directly in the background.
  • CTRL+C: Stops the current foreground job.
  • CTRL+Z: Pauses the current foreground job.
  • bg: Moves a paused job to the background.
  • fg: Brings a background job to the foreground.

Foreground Process

user@lfs101: ~
$ sleep 10
In this case, you have to wait 10 seconds to use your terminal again.

Background Process

user@lfs101: ~
$ sleep 100 &amp;
[1] 12345
$ sudo apt update
In this case, you can continue use the terminal. [1] is Job ID and 12345 is Process ID.

Managing Jobs

The jobs command lists all jobs currently running in the background of your terminal.

  • jobs: Shows the Job ID, status (e.g., running, stopped), and command name.
  • jobs -l: Shows the same information plus the Process ID (PID).

Important: Background jobs are tied to your terminal session. If you close the terminal or log out, these jobs will be terminated.

user@lfs101: ~
$ sleep 100 &amp;
[1] 6440
$ sleep 100 &amp;
[2] 6450
$ jobs
[1] Running           sleep 100 &amp;
[2] Running           sleep 100 &amp;
$ sleep 100 &amp;
[3] 6460
$ jobs -l
[1] 6440 Running           sleep 100 &amp;
[2] 6450 Running           sleep 100 &amp;
[3] 6460 Running           sleep 100 &amp;

Listing Processes: ps and top

The ps Command (System V Style)

The ps command gives you a snapshot of currently running processes, which are identified by a Process ID (PID). For a live, auto-updating view, it's better to use tools like top or htop.

The ps command has many options, but here are the most common ones:

  • ps: Shows only the processes you are running in your current terminal.
  • ps -u username: Lists the processes owned by a specific user.
  • ps -ef: Shows every single process running on the entire system.
  • ps -eLf: Goes even deeper, showing every thread within every process.
user@lfs101: ~
$ ps
    PID TTY          TIME CMD
  66133 pts/0    00:00:00 bash
  66502 pts/0    00:00:00 ps

The ps Command (BSD Style)

The ps command also accepts an older style of options that don't use a dash (-). This is known as the "BSD style".

  • ps aux: A very common command that, like ps -ef, shows every process running on the system.
  • ps axo: Lets you create a custom view by specifying exactly which columns (attributes) you want to see, such as the PID or command name.
user@lfs101: ~
$ ps aux | axo

Top

While ps gives you a single snapshot, the top command provides a live, real-time view of your system's performance. 📟

It automatically updates every few seconds, making it easy to see which processes are using the most CPU and memory at any given moment. To exit top, simply press the q key.

user@lfs101: ~
$ top

First Line of The top Output

The first line of the top output displays a quick summary of what is happening in the system, including:

  • How long the system has been up
  • How many users are logged on
  • What is the load average

Second Line of The top Output

The second line in top (labeled "Tasks") provides a quick count of all processes and their current status: running, sleeping, stopped, or zombie.

Third Line of The top Output

The third line in top (labeled "%Cpu(s)") shows a detailed breakdown of how your CPU is spending its time, with each value as a percentage. 📊

  • us (user): Time spent running user applications.
  • sy (system): Time spent on core operating system tasks.
  • ni (nice): Time spent on low-priority user tasks.
  • id (idle): Time the CPU is unused. A high percentage means the system is not busy.
  • wa (I/O wait): Time spent waiting for tasks like reading from or writing to the disk.
  • hi/si (interrupts): Time spent handling hardware (hi) and software (si) requests.
  • st (steal time): In a virtual machine, this is time the host machine took from the guest CPU.

Fourth and Fifth Lines of The top Output

The fourth and fifth lines of the top output indicate memory usage, which is divided in two categories:

  • Physical memory (RAM) – displayed on line 4.
  • Swap space – displayed on line 5.

Both categories display total memory, used memory, and free space.

For your system to run fast, it's crucial to have enough physical memory (RAM).

When the RAM is full, your system starts using a part of the hard drive called swap space as emergency memory. Because hard drives are much slower than RAM, this will cause your system's performance to drop significantly.

Process List of The top Output

Each line in the process list of the top output displays information about a process. By default, processes are ordered by highest CPU usage. The following information about each process is displayed:

  • Process Identification Number (PID)
  • Process owner (USER)
  • Priority (PR) and nice values (NI)
  • Virtual (VIRT), physical (RES), and shared memory (SHR)
  • Status (S)
  • Percentage of CPU (%CPU) and memory (%MEM) used
  • Execution time (TIME+)
  • Command (COMMAND).

Interactive Keys With top

The top command is also an interactive tool that you can control with single-key commands while it is running. ⌨️

For example, you can:

  • Change the sorting to see which processes are using the most memory or CPU.
  • Adjust the priority of a running process.
  • Kill a process that is misbehaving or needs to be stopped.

If you want a more modern and user-friendly tool, there are popular alternatives to top. Check out programs like htop, btop, and atop, which often have better visuals and more features.

Starting Processes In The Future

Scheduling Future Processes Using at

If you need to schedule a command to run automatically at a specific time in the future (especially when you'll be away from the computer), you can use the at command.

user@lfs101: ~
$  at now + 2 days
warning: commands will be executed using /bin/sh
at Wed Oct  8 15:16:00 2025
at&gt;

cron

The cron utility is a scheduler for automating recurring tasks, like running a backup every night or checking for updates every week. This is different from the at command, which runs a task only once.

cron works by reading a configuration file called a crontab. Each line in this file defines a single job, specifying the schedule (as a CRON expression) and the command to run.

To edit your personal list of scheduled jobs, you use the command crontab -e.

FieldDescriptionValues
MINMinutes0 to 59
HOURHour field0 to 23
DOMDay of Month1-31
MONMonth field1-12
DOWDay of Week0-6 (0 = Sunday)
CMDCommandAny command to be executed

Examples:

  • The entry * * * * /usr/local/bin/execute/this/script.sh will schedule a job to execute script.sh every minute of every hour of every day of the month, and every month and every day in the week.
  • The entry 30 08 10 06 * /home/sysadmin/full-backup will schedule a full-backup at 8.30 a.m., 10-June, irrespective of the day of the week.

anacron

Modern Linux systems often use anacron as an improvement over the classic cron.

The main problem with cron is that if your computer is turned off when a job is scheduled, that job is simply skipped. anacron solves this by checking for missed jobs the next time the system starts up and running them then.

This makes anacron ideal for desktops and laptops that aren't running 24/7. Its main configuration file is /etc/anacrontab

sleep

The sleep command is used to pause the execution of a script or command for a specific amount of time. This is useful when you need to wait for a resource to become available or to create a periodic loop.

You tell sleep how long to wait by giving it a number and an optional suffix:

  • s: seconds (the default)
  • m: minutes
  • h: hours
  • d: days

Important: Don't confuse sleep with at. sleep pauses for a duration (e.g., "wait for 5 minutes"), while at schedules a job to run at a specific time (e.g., "run at 2:30 PM").

Summary

Understanding Processes

A process is an instance of a running program that performs tasks on the system. Processes can be single or multi-threaded and can run interactively (with a user) or non-interactively (in the background).

  • Process ID (PID): Every process is assigned a unique integer ID so the operating system can manage it.
  • nice value: A number that can be adjusted to raise or lower a process's scheduling priority.

Monitoring System Activity

Several tools are available to monitor what your system is doing.

  • ps: Provides a static snapshot of the currently running processes.
  • top: Gives a constant, real-time, interactive view of overall system performance and the processes consuming the most resources.
  • Load Average: A set of three numbers that indicates how busy the system has been over the last 1, 5, and 15 minutes.

Job Control & Scheduling

You can control how and when commands are executed.

  • Background & Foreground: Jobs can run in the foreground, tying up your terminal, or be sent to the background to run independently while you continue to work.
  • at: A command used to schedule a task to be run once at a specified time in the future.
  • cron: A scheduler used for tasks that need to be performed repeatedly at regular intervals (e.g., running a backup every night at 2 AM).
