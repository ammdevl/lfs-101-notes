Local Security Principles

Understanding Linux Security

User Accounts

In Linux, user accounts are used to control access to files and applications. The system identifies users in two ways and provides standard tools for account management.

How Users are Identified

Each user account has two main identifiers:

  • A human-readable username (like john).
  • A unique number called a UID (user ID) that the system uses internally.

A central user database links each username to its corresponding UID.

The Account Creation Process

When a new user is created, the system performs several key actions:

  1. Adds the new user's information to the user database.
  1. Creates a personal home directory for the user.
  1. Populates the home directory with a set of default configuration files.

This process can be managed using either graphical (GUI) applications or standard command-line tools like useradd (to create an account) and userdel (to remove one).

Types of Accounts

Linux uses different account types to securely separate processes and workloads. Managing these accounts with good security practices is crucial, especially in a multi-user environment.

Four Types of Accounts

To organize permissions and isolate tasks, Linux defines four main types of accounts:

  • root: The superuser with complete administrative access to the entire system.
  • System: Accounts used by the operating system to run services in the background (e.g., the www-data user for a web server). These are not meant for human login.
  • Normal: Standard user accounts for people to log in and perform daily tasks.
  • Network: Accounts used for network services.

Key Security Principles

For a safe working environment, two main principles should be followed:

  1. Principle of Least Privilege: Grant accounts only the minimum privileges they need to perform their function.
  1. Account Housekeeping: Remove inactive accounts to reduce potential security risks. You can use the last command to check when users last logged in to help identify unused accounts.

Context Matters: Enterprise vs. Personal Systems

Security practices are naturally much stricter on multi-user enterprise servers than they might be on a single-user personal computer. While it's best to learn and apply enterprise-level security standards, users may choose to relax some of these rules on their own personal systems where the risks are different.

Understanding The root Account

The Power of root

The root account is the superuser in Linux, possessing complete and unrestricted control over the entire operating system. It's like having the master key to everything on the system. This account has no security restrictions and can perform any administrative task, including creating users, installing software, and modifying any file. Because of this, it must be used with extreme caution.

The Warning Sign: The # Prompt

To serve as a constant reminder of this absolute power, the default command-line prompt for the root user changes from a dollar sign ($) to a # (hash sign).

This visual cue is a built-in warning to be extra careful, as any command executed as root can have immediate and system-wide consequences.

When Are root Privileges Required?

Operations Requiring root Privileges

In Linux, there is a clear separation between everyday user tasks and administrative tasks that can affect the entire system. Performing system-wide administrative actions requires root privileges (also known as superuser or administrator privileges).

Think of it like living in an apartment building: a regular user can decorate and manage their own apartment, but only the building superintendent (root) can make changes to the building's core infrastructure.

Tasks Requiring root Privileges

You will need root privileges (usually obtained via the sudo command) to perform tasks that impact the whole system, such as:

  • Managing user accounts (creating, deleting, or modifying users).
  • Installing, removing, and upgrading software system-wide.
  • Modifying system files located outside of your personal home directory (e.g., files in /etc or /usr).
  • Starting, stopping, and restarting system services (daemons).

🗝️ Tasks for Regular Users

Regular users have full control over the files and directories within their own home directory. They are generally allowed to customize their personal settings and run applications. While they can sometimes install software through graphical "app stores," these tools handle the root authentication in the background; the user themselves cannot directly modify system-wide application directories.

Operations Not Requiring root Privileges

🗝️ What is SUID?

SUID (Set User ID) is a special file permission that applies only to executable files. When a program with the SUID bit is run, it executes with the permissions of the file's owner, rather than the user who is running it.

This grants temporary permissions to the user, but only for the duration and scope of that specific program's execution. It's similar in concept to the "run as" feature in Windows.

The root Use Case

The most significant use of SUID is for programs owned by the root user. When a regular user runs an SUID-enabled program owned by root, the program executes with root privileges.

This is a controlled way to allow regular users to perform specific administrative tasks without giving them a full root shell or sudo access. A classic example is the passwd command, which allows users to change their own password by temporarily gaining the root permissions needed to modify protected system files like /etc/shadow.

sudo Features

The sudo command includes important features for security auditing and is controlled by specific configuration files.

Logging & Auditing

A key security feature of sudo is its ability to log all usage attempts, especially unsuccessful ones. When an unauthorized user tries to run a command with sudo, the event is recorded in a system log file, which is typically /var/log/secure. This allows system administrators to monitor for and investigate suspicious activity.

Configuration

The rules that determine which users are allowed to use sudo and what commands they are permitted to run are stored in configuration files.

  • The primary configuration file is /etc/sudoers.
  • Additional configuration rules can be placed in separate files inside the /etc/sudoers.d/ directory.

The sudoers’ Files

The sudo command's behavior is controlled by rules defined in configuration files. To prevent accidental system damage, these files must be edited with a special, safe command called visudo.

Sudo Configuration Files

When sudo is run, it checks for rules in two places:

  • The main configuration file: /etc/sudoers
  • Any file inside the directory: /etc/sudoers.d/

The basic structure for a rule is who where = (as_whom) what. The modern best practice is to add user-specific rules in a new file inside the /etc/sudoers.d/ directory, rather than modifying the main /etc/sudoers file directly.

The visudo Command: The Safe Way to Edit

You must always use the visudo command to edit these configuration files. Using a normal text editor like nano or vi is dangerous.

visudo provides critical safety features, the most important of which is syntax checking. It will parse the file for errors before saving and will refuse to save your changes if it finds a mistake. This prevents you from accidentally breaking the sudo configuration, which could lock you out of administrative access to your system.

  • To edit the main file, simply run: sudo visudo
  • To edit a specific file in the .d directory, use the f flag: sudo visudo -f /etc/sudoers.d/your_file

Command Logging

For security and accountability, the sudo command logs every action that is attempted, including both successful commands and failed attempts.

What is Logged?

Each log entry acts like a detailed receipt, recording crucial information about the event:

  • Who: The username of the person who ran the command.
  • When: The exact date and time.
  • Where: The terminal and working directory from which it was run.
  • What: The exact command that was executed, including all arguments.

Where are the Logs?

The location of the sudo log file varies depending on your Linux distribution family:

  • Debian/Ubuntu: /var/log/auth.log
  • Red Hat/Fedora/SUSE: /var/log/secure (or sometimes /var/log/messages).

Process Isolation

Linux's security model is built on the fundamental principle of process isolation, which is enhanced by several modern technologies to create multiple layers of defense.

The Foundation: Process Isolation

By default, Linux is considered secure because processes are naturally isolated from one another. This means one running program cannot normally access the memory or resources of another program, even if they are run by the same user. This core feature makes it difficult for viruses and malware to spread and attack other parts of the system.

Advanced Isolation Technologies

Modern Linux builds upon this foundation with even more powerful isolation tools, creating layers of security.

  • Control Groups (cgroups): This is a kernel feature that allows an administrator to group processes together and, crucially, to limit the system resources (like CPU time, memory, or network bandwidth) that the group can consume.
  • Containers: Built on top of cgroups and other kernel features, container technology allows you to run applications and their dependencies in lightweight, isolated user-space instances. Each container runs on the same shared kernel but is walled off from the other containers and the host system.
  • Virtualization: This provides the highest level of isolation by emulating an entire hardware system. It allows you to run multiple, completely separate operating systems (known as virtual machines or guests) on a single physical computer (the host). Each virtual machine has its own kernel and is fully insulated from the others.

Hardware Device Access

Linux has a simple and powerful model for securing access to hardware: it treats devices as if they were files.

How it Works

For every piece of non-networking hardware connected to your system (like a hard drive, webcam, or USB device), the kernel creates a special file called a device node in the /dev directory. Applications interact with the hardware by reading from and writing to this special file.

How this Enforces Security

Because these device nodes are treated like regular files, they have the same standard file permissions system (owner, group, and others with read, write, and execute permissions).

This provides a natural and consistent security model. By default, regular users do not have permission to write directly to a raw hardware device like a hard disk (/dev/sda1). This is a critical safeguard, because direct, low-level access is extremely dangerous and can instantly destroy an entire filesystem.

Normal file operations (like saving a document) are handled safely by the high-level filesystem, not by directly accessing these low-level device nodes.

Keeping Current

One of the most critical aspects of maintaining a secure system is applying software updates in a timely manner.

How Linux Updates Work

Linux distributions have an excellent track record of responding quickly to security vulnerabilities, bug fixes, and performance improvements. When a problem is discovered, a fix (or "patch") is promptly released through the distribution's official software repositories, and users are notified that an update is available.

The Danger of Delay

The biggest security risk to most computer systems is not a new, unknown threat, but rather an old vulnerability for which a fix has already been released but has not been applied. Many of the most successful cyberattacks exploit these known, unpatched security holes. Users sometimes hesitate to update, fearing that a patch might break their system, but this is a much bigger problem on proprietary operating systems.

The Best Practice: Update Promptly

The best and simplest way to keep your system secure is to take advantage of your Linux distribution's update mechanism.

You should enable automatic updates whenever possible and never postpone them. It is extremely rare for a standard update on a stable Linux distribution to cause new problems, and the security benefits of staying current far outweigh the minimal risk.

Working With Passwords

How Passwords Are Stored

The system verifies authenticity and identity using user credentials.

Originally, encrypted passwords were stored in the /etc/passwd file, which was readable by everyone. This made it rather easy for passwords to be cracked.

Password Encryption

Modern password protection is a critical part of system security. Most Linux distributions have moved away from older, weaker methods and now rely on a strong cryptographic hashing algorithm to keep passwords safe.

The Hashing Algorithm: SHA-512

The standard algorithm used by most modern Linux systems to protect passwords is SHA-512 (Secure Hashing Algorithm 512 bits). It was developed by the U.S. National Security Agency (NSA) and is a heavily tested, widely trusted algorithm used in many other security protocols, including TLS/SSL and SSH.

How Hashing Works

Password protection in Linux uses hashing, which is a one-way process. The system does not store your actual password. Instead, it runs your password through the SHA-512 algorithm to produce a unique, fixed-length string of characters called a hash.

When you log in, the system hashes the password you type and compares that new hash to the one it has stored. Because the process is one-way, it's computationally infeasible to reverse the process and figure out the original password from the stored hash. You can experiment with this yourself using the sha512sum command-line utility.

Good Password Practices

IT professionals employ several strategies to enhance password security, moving from reactive measures to proactive enforcement and auditing.

Password Aging

Password aging is a policy that forces users to change their passwords after a specific period of time. This limits the window of opportunity for an attacker if a password is stolen or cracked. In Linux, this is configured for each user with the chage command.

Enforcing Strong Passwords

This practice involves configuring the system to automatically reject weak passwords when a user tries to set a new one. This is implemented using PAM (Pluggable Authentication Modules), which act as a gatekeeper during the password change process.

PAM uses a library like pam_cracklib.so or the more modern pam_passwdqc.so to check if a new password meets complexity requirements (e.g., length, character variety).

Proactive Auditing

This is a defensive technique where a system administrator uses a password cracking program, such as John The Ripper, to actively try to crack the passwords of users on their own system. The goal is to identify and remediate weak passwords before an attacker can find them.

Important: You must obtain explicit written authorization before running password cracking tools on any system that you do not personally own.

Securing the Boot Process and Hardware Resources

Requiring Boot Loader Passwords

You can enhance your system's security by setting a password on the GRUB bootloader, but this is only effective when combined with other security measures.

Securing the Boot Process

Setting a bootloader password on GRUB prevents an unauthorized user with physical access from easily modifying boot parameters to bypass security (for example, by booting into single-user mode).

However, a GRUB password alone will not prevent someone from booting the machine with an external device like a USB drive. For complete protection, a bootloader password must be used in conjunction with a BIOS/UEFI password that is configured to disable booting from external media.

Configuring GRUB 2

Configuring the modern GRUB 2 bootloader is more complex than its predecessor but offers more flexibility. The most important rule is to never edit the /boot/grub/grub.cfg file directly, as your changes will be overwritten.

The correct, two-step process is:

  1. Make your changes to the configuration template files, primarily /etc/default/grub and the scripts in the /etc/grub.d/ directory.
  1. Run the command update-grub (or grub2-mkconfig on some systems) to apply your changes and generate a new grub.cfg file.

Hardware Vulnerability

Physical security is the foundational layer of computer security. If an attacker can physically access a computer, many software-level protections become irrelevant. Your IT security policy must therefore begin with securing the hardware itself.

Physical Threats

When hardware is physically accessible, it opens up several major attack vectors that can bypass the operating system's security:

  • Key Logging: An attacker can install a hardware device to record every keystroke, capturing passwords and other sensitive data.
  • Network Sniffing: Data packets can be captured directly from unprotected network cables or switches.
  • Booting from a Live Disk: An attacker can boot the computer from their own USB drive or DVD, allowing them to bypass the installed operating system and directly access and modify the hard drive's contents.

Essential Countermeasures

To defend against these threats, the following physical security guidelines are essential:

  • Lock Down Hardware: Servers and workstations should be kept in physically secure, locked rooms.
  • Protect Network Links: Network cables, switches, and routers should be in secure locations where they cannot be tampered with.
  • Set a BIOS/UEFI Password: This is a critical step. You should set a password on the system's BIOS/UEFI and configure it to prevent booting from external devices like USB drives or optical disks. This is the primary defense against live disk attacks.

Home vs. Enterprise Context

While these strict measures are mandatory for business servers, they might be considered excessive for a typical home computer. However, if your personal system contains highly sensitive information, you should follow these stricter guidelines to ensure it is properly protected.

Software Vulnerability

Like any complex software, security vulnerabilities are sometimes found in the Linux ecosystem. However, a major strength of the Linux and open source community is the speed at which these issues are addressed.

When a weakness is discovered, it is typically exposed, fixed, and a patch is released very quickly, often much faster than in many proprietary software environments.

Summary

Linux security is built on a multi-layered approach, from the operating system's core design to administrative best practices.

Managing Privileges: root & sudo

The root account is the superuser, with unlimited power to modify the entire system. Because this level of access is so dangerous for routine use, the sudo command is the preferred method for performing administrative tasks.

  • sudo grants elevated privileges on a temporary, per-command basis.
  • Its rules are defined in the /etc/sudoers file (and /etc/sudoers.d/ directory).
  • For security and accountability, sudo logs all commands and failed attempts to system log files like /var/log/auth.log or /var/log/secure.

Password & Authentication Security

Linux uses strong methods to protect user credentials.

  • Password Hashing: Passwords are not stored in a recoverable format. They are protected using a one-way hashing function, typically the SHA-512 algorithm.
  • Strong Password Policies: PAM (Pluggable Authentication Modules) can be configured to enforce password complexity rules, forcing users to create strong, hard-to-guess passwords.

Foundational System Security

Beyond user privileges, several core principles are essential for a secure system.

  • Process Isolation: A fundamental security feature of the Linux kernel is that processes are naturally isolated and cannot access each other's resources, even when run by the same user.
  • Physical Security: The most basic and critical layer of security is protecting physical access to servers and workstations. If an attacker can touch a machine, many software-level protections can be bypassed.
  • System Updates: One of the most important steps in avoiding security breaches is to keep the system's software regularly patched and updated to protect against known vulnerabilities.
