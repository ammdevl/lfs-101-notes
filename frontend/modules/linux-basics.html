Linux Basics and System Setup

The Boot Process

The Linux boot process is the procedure for initializing the system. It consists of everything that happens from when the computer power is first switched on until the user interface is fully operational.

BIOS - The First Step

While Linux runs on many kinds of hardware, we will concentrate on the x86 family, which is the basis of almost all desktop and laptop PCs. Starting an x86-based Linux system involves a number of steps. When the computer is powered on, the Basic Input/Output System (BIOS) initializes the hardware, including the screen and keyboard, and tests the main memory. This process is also called POST (Power On Self Test).

The BIOS software is stored on a read-only memory (ROM) chip on the motherboard. After this, the remainder of the boot process is controlled by the operating system (OS).

Master Boot Record (MBR), EFI Partition, and Boot Loader

After POST, control passes from BIOS to the boot loader, which is stored on a storage device like a hard disk or SSD in either the boot sector (BIOS/MBR systems) or EFI partition (UEFI systems). Until this point, the machine doesn't access mass storage. Date, time, and key peripheral information are loaded from CMOS values (battery-powered memory that tracks date and time when powered off).

Common Linux boot loaders include GRUB, ISOLINUX (for removable media), and DAS U-Boot (for embedded devices). These boot loaders typically offer a user interface for selecting boot options for Linux and other installed operating systems. The boot loader's primary function is loading the kernel image and initial RAM disk/filesystem (containing critical files and drivers) into memory.

The Boot Loader In Action

The boot loader has two distinct stages:

For BIOS/MBR systems, the boot loader sits in the Master Boot Record (MBR)—a 512-byte section at the disk's first sector. It examines the partition table, finds a bootable partition, then loads the second-stage loader (like GRUB) into RAM. In EFI/UEFI systems, the firmware reads its Boot Manager data to determine which UEFI application to launch and from where. It then launches the application (e.g., GRUB) as defined in the boot entry—a more complex but flexible approach than MBR methods.

The second-stage loader resides in /boot and displays a splash screen for OS/kernel selection. Once selected, it loads the kernel into RAM and transfers control. The compressed kernel first decompresses itself, then analyzes hardware and initializes device drivers.

The Initial RAM Disk

The initramfs filesystem image contains programs and binary files needed to mount the proper root filesystem. It provides kernel functionality for the specific filesystem and loads device drivers for mass storage controllers using udev, which identifies present devices, locates necessary drivers, and loads them. Once found, the root filesystem is checked and mounted.

The mount program tells the OS that a filesystem is ready and associates it with a mount point in the filesystem hierarchy. Upon success, initramfs is cleared from RAM, and the init program (/sbin/init) executes.

init handles mounting and pivoting to the final root filesystem. Any special hardware drivers needed to access mass storage must be in the initramfs image.

Text-Mode Login

Near the end of the boot process, init launches text-mode login prompts where users enter credentials to access a command shell. These aren't visible with graphical login interfaces.

In Command Line Operations, command shells run in terminals accessed via ALT + function keys. Most distributions provide six text terminals and one graphics terminal. From a graphical environment, access text consoles with CTRL-ALT + function key (F7 or F1 returns to GUI).

The default shell is typically bash (GNU Bourne Again Shell), though other shells exist. The shell displays a prompt for commands, executes them when you press Enter, and shows another prompt afterward.

Kernel, init and Services

The Linux Kernel

The boot loader loads both the kernel and initial RAM-based file system (initramfs) into memory for direct kernel use.

Once loaded, the kernel initializes memory and configures all hardware components including processors, I/O subsystems, and storage devices, then loads essential user space applications.

/sbin/int and Serivces

After hardware setup and root filesystem mounting, the kernel runs /sbin/init, which becomes the initial process that starts all other system processes. Most processes trace back to init, except kernel processes which manage internal OS details.

init maintains system operation and handles clean shutdowns. It manages non-kernel processes, cleaning up after completion and restarting services as needed when users log in/out.

Traditionally, system startup followed 1980s System V UNIX conventions. This SysVinit method used sequential runlevels with scripts to start and stop services. Each runlevel supported different operating modes.

Major distributions have now moved away from this sequential initialization, though they maintain System V compatibility. The newer systemd method has become the dominant replacement.

Systemd Features

systemd enables faster startup than earlier init methods by using parallelization instead of sequential steps, allowing simultaneous service initiation.

Simple configuration files replace complex startup scripts, defining service prerequisites, execution methods, and completion conditions. Note that /sbin/init now points to /lib/systemd/systemd, effectively making systemd the init process.

Most basic tasks use a single command (systemctl):

  • Start/stop/restart a service:
user@lfs101: ~
$ sudo systemctl start|stop|restart httpd.service | apache2
  • Enable/disable at boot:
user@lfs101: ~
$ sudo systemctl enable|disable httpd.service | apache2
  • Check service status:
user@lfs101: ~
$ sudo systemctl status httpd.service | apache2

The .service suffix is typically optional.

Linux Filesystem Basics

Linux Filesystems

A filesystem organizes and stores data in a usable form, similar to how a library organizes books. It's a method for structuring and managing collections of data.

Linux supports these filesystem types:

  • Conventional disk filesystems: ext3, ext4, XFS, Btrfs, JFS, NTFS, vfat, exfat, etc.
  • Flash storage filesystems: ubifs, jffs2, yaffs, etc.
  • Database filesystems
  • Special purpose filesystems: procfs, sysfs, tmpfs, squashfs, debugfs, fuse, etc.

Linux Installation

Planning

It's best to decide on partition layout during installation as changing it later can be difficult. Linux systems mount multiple partitions at specific points in the filesystem, and while you can modify this setup later, getting it right from the start is much easier.

Most Linux distribution installers offer sensible default layouts. These typically include either one large partition for all normal files plus a smaller swap partition, or separate partitions for space-sensitive areas like /home and /var. You might need to customize this arrangement if you have special requirements or plan to use multiple disks.

Software Choices

All installations include the minimum software needed to run Linux.

Distributions offer additional software categories including common applications (Firefox, LibreOffice), developer tools (vi, emacs), popular services (Apache, MySQL), and for graphical systems, a default desktop environment (GNOME or KDE).

Modern distributions perform a quick basic install first, then let users add software later. This improves upon earlier approaches that required many upfront decisions, which often confused new users and extended installation time.

Installers configure basic security features, including setting the root password and creating an initial user. Some distributions (like Ubuntu and Fedora) disable direct root login, requiring sudo access through a regular user account. Advanced security frameworks may be installed by default: SELinux on Red Hat-based systems and AppArmor on Ubuntu.

Install Source

Linux distributions are available on USB drives, CDs, or DVDs. Most also support booting from a small image and downloading remaining components over the network. These images can be used on physical media or as network boot images, enabling installation without local media.

Many installers support fully automated installation using configuration files: Kickstart (Red Hat), AutoYAST (SUSE), or Preseed (Debian).

Each distribution offers documentation and tools for managing these configuration files.

The Process

The installation process is similar across distributions. After booting from installation media, the installer asks system configuration questions (unless using an automatic installation file), performs the installation, then reboots into the new system for final setup.

Most installers can download and install updates during installation if Internet access is available. Otherwise, updates are retrieved later through the system's normal update mechanism.

The Alternative Methods

  • Re-partitioning your hard disk to free up enough room to permit dual boot (side-by-side) installation of Linux, along with your present operating system.
  • Using a host machine hypervisor program (such as VMWare's products or Oracle Virtual Box) to install a client Linux Virtual Machine.
  • Booting off of and using a Live CD or USB stick and not writing to the hard disk at all.

Summary

Disk Organization: Partitions & Filesystems

A hard disk is logically divided into sections called partitions. Within each partition, a filesystem is created, which is the structure used to store, organize, and find files.

The main benefit of this approach is data protection. By separating different types of data (e.g., system files vs. user files) onto different partitions, an error or failure in one partition is less likely to damage the data in the others.

The Boot Process

When a Linux system starts up, it follows a multi-step boot process:

  1. The BIOS (or UEFI) performs initial hardware checks and starts the process.
  1. It then triggers the boot loader (like GRUB).
  1. The boot loader loads the Linux kernel into memory and starts it.
  1. The kernel uses an initramfs (initial RAM filesystem) to load necessary drivers.
  1. Finally, the init program (like systemd) is started, which brings up the rest of the system services and the user login screen.

There is no single "best" Linux distribution. The right choice depends on your specific needs. To select the appropriate distribution, you should match your requirements (e.g., for a server, desktop, or embedded device) with the features, software, and support offered by different distributions like Fedora, Ubuntu, or openSUSE.
